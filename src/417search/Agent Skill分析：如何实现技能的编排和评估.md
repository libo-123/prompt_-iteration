# Agent Skill分析，如何实现技能的编排和评估

链接：https://zhuanlan.zhihu.com/p/2014393687812110022

## 一句话总结

这篇文章的核心观点是：**Agent Skills 正在从“零散经验”变成一套可管理、可编排、可评估的基础设施。**

过去大家更多关注“模型够不够强”，而这几篇论文共同说明，未来 Agent 的关键竞争力，可能不只是模型能力，而是：**能不能把技能沉淀下来、组织起来、调度起来，并验证这些技能真的有效。**

## 通俗理解：这 3 篇论文分别解决什么问题

可以把它们理解成 Agent Skill 体系里的三个层次：

- `SkillOrchestra`：解决“**当前这一步该让谁来做**”的问题，重点是技能驱动的路由与调度。
- `AgentSkillOS`：解决“**技能多起来以后怎么管理、怎么找、怎么组合**”的问题，重点是生态级技能管理与工作流编排。
- `SkillsBench`：解决“**技能到底有没有用、什么时候最有用**”的问题，重点是评测和方法论。

如果用公司团队来打比方：

- `SkillOrchestra` 像项目经理，决定这一步该派谁上。
- `AgentSkillOS` 像组织架构和流程系统，负责管理公司能力并把多人协作串起来。
- `SkillsBench` 像绩效考核体系，验证这些 SOP 和能力包到底有没有提升结果。

## 1. SkillOrchestra：不要按模型路由，要按技能需求路由

这篇论文最重要的洞察是：

**Agent 编排不应该只看哪个模型更强，而应该看当前步骤到底需要什么技能。**

### 以前的问题

传统方法通常有两个问题：

- 只在用户输入时做一次粗粒度路由，无法适应任务中途需求变化。
- 用强化学习学习编排策略，训练成本高，而且容易“路由坍缩”，最后大量请求都堆给最强最贵的模型。

### 它的做法

SkillOrchestra 提出了一个 `Skill Handbook`（技能手册）的概念。可以把它理解为一本“调度说明书”，里面记录：

- 当前任务处于什么操作模式，比如搜索、推理、写代码；
- 每种模式需要哪些技能；
- 不同 Agent 在这些技能上的表现和成本。

系统在每一轮交互中做两步决策：

1. 先判断当前属于什么模式；
2. 再识别当前步骤需要哪些技能，并综合性能和成本，选择最合适的 Agent。

### 通俗理解

它本质上是把 Agent 编排从“拍脑袋选最强模型”，升级成“按工种派活”。

### 这篇论文的价值

- 性能比现有强化学习编排方法更好；
- 成本更低；
- 不容易把所有任务都挤到单一强模型上；
- 技能手册还能迁移到别的模型上继续使用。

这说明：**技能层是比单一模型更稳定、更可复用的中间抽象。**

## 2. AgentSkillOS：技能多了以后，必须像操作系统一样管理

这篇论文讨论的是一个更现实的问题：

**当技能变成几万、几十万级别时，Agent 该怎么高效找到、管理并组合这些技能？**

### 它要解决的痛点

随着 Skill 生态快速增长，出现了三个典型问题：

- 技能太多，用户找不到真正适合任务的技能；
- 技能质量参差不齐，平台治理难；
- 扁平式调用方式难以把多个技能真正组合起来，技能越多越难用。

### 它的核心思路

AgentSkillOS 把整个问题拆成两个阶段：

### 第一阶段：管理技能

它先构建一个 `能力树`（Capability Tree），把海量技能按层级组织起来。

你可以把它理解成一个技能版“目录系统”：

- 顶层先按大类分组；
- 再逐级细分；
- 最终把具体技能挂到合适的位置。

这样做的好处是：

- 技能检索不再需要全量扫描；
- 新技能可以被持续插入和更新；
- 大规模技能池也能保持结构化和可用性。

### 第二阶段：求解任务

在找到候选技能后，系统不会简单顺序调用，而是把多个技能编排成一个 `DAG`（有向无环图）执行计划，自动处理：

- 哪些步骤先做；
- 哪些步骤可以并行；
- 哪些步骤依赖上一步结果；
- 数据如何在多个技能之间流动。

它还支持三种不同目标的编排策略：

- `质量优先`：追求最好结果；
- `效率优先`：追求更高并行度和更快执行；
- `极简优先`：只保留必要节点，减少复杂度。

### 通俗理解

如果说 SkillOrchestra 解决的是“这一步该派谁”，那么 AgentSkillOS 解决的是“整个项目流程怎么排”。

它的意义在于：**技能不再是零散插件，而是可以被组织、检索、编排、复用的生产系统。**

## 3. SkillsBench：技能确实有用，但不是越多越好

这篇论文的意义在于，它第一次比较系统地回答了：

**Skill 到底值不值得做？**

答案是：**值得，但效果高度依赖设计方式和使用场景。**

### 最核心的发现

#### 1. 人工精选技能确实有效

人工精选的 Agent Skills 平均可以把任务通过率提升 `16.2%`。  
这说明 Skill 不是概念包装，而是真的能提升任务完成效果。

#### 2. 但效果不是通用的

技能的收益高度依赖：

- 任务所在领域；
- 使用的模型；
- 所在的 Agent 框架；
- 技能本身的设计质量。

也就是说，不是“有 Skill 就一定更强”，而是“**合适的 Skill 在合适场景中才会显著增益**”。

#### 3. 模型自己生成 Skill，暂时并不靠谱

论文发现，模型自生成技能平均反而让通过率下降 `1.3%`。  
这说明高质量 Skill 的关键仍然是**人类专家的过程性经验**，而不是模型自动写一篇说明文档。

#### 4. 技能设计要遵循“少即是多”

这是最有指导意义的一点：

- 1 个技能已经有明显增益；
- 2-3 个技能时效果最好；
- 4 个及以上技能时，收益明显下降。

原因是技能太多会导致**认知过载**，让 Agent 被过量信息干扰。

同时，论文也发现：  
**中等长度、聚焦明确、步骤清晰的技能效果最好；冗长的百科式技能反而可能拖后腿。**

### 通俗理解

Skill 对 Agent 来说，不是一本越厚越好的百科全书，而是关键时刻拿出来就能用的简明 SOP。

## 这三篇论文合起来说明了什么

如果把三篇论文放在一起看，可以得到三个非常清晰的趋势：

### 1. Skill 正在成为 Agent 的核心中间层

Skill 不再只是 Prompt 的补充材料，而是：

- 路由调度的依据；
- 能力组织的单位；
- 多技能协作的节点；
- 可评估、可迁移、可复用的能力资产。

### 2. Agent 的竞争，正在从“模型能力”转向“技能系统能力”

未来真正重要的不只是模型强不强，还包括：

- 技能能不能被准确描述；
- 技能能不能被快速检索；
- 多技能能不能被合理编排；
- 技能效果能不能被可靠评估。

换句话说，未来 Agent 的壁垒，很可能是**围绕 Skill 搭建出来的能力操作系统**。

### 3. 好的 Skill 更像产品，而不是一堆文档

SkillsBench 给出了很明确的设计启发：

- 不是越多越好；
- 不是越长越好；
- 不是自动生成就够了；
- 而是要聚焦、模块化、任务相关、步骤清晰。

高质量 Skill 更像一个可以直接拿来落地的能力包，而不是一篇知识大而全的说明书。

## 最后可以怎么概括这篇文章

如果要用一段最通俗的话来总结，可以写成：

> 这几篇关于 Agent Skills 的论文共同说明，Agent 正在从“依赖单模型能力”走向“依赖技能系统能力”。  
> `SkillOrchestra` 解决的是如何按技能需求选择合适的 Agent，`AgentSkillOS` 解决的是如何在大规模技能生态中组织和编排技能，`SkillsBench` 则证明了高质量技能确实能提升 Agent 表现，但前提是技能必须足够聚焦、精简，并且来自真实的人类专业经验。  
> 未来 Agent 的关键竞争力，可能不只是模型更强，而是谁更能把技能沉淀、管理、编排和验证成一套真正可复用的能力体系。

## 可直接提炼的 3 个观点

1. **Skill 不再只是 Prompt 补丁，而是 Agent 的能力中间层。**
2. **技能的价值不在数量，而在能否被准确检索、合理编排、有效执行。**
3. **高质量 Skill 依然高度依赖人类专家经验，2-3 个聚焦技能往往比一堆冗长说明更有效。**
