# PRD 分析 Prompt

## 使用说明

用于生成技术架构和开发任务列表。使用 AI IDE 工具分析代码结构，不需要执行脚本。

## 执行流程

### 1. 依赖检查

**检查并安装依赖 Skills**：

| Skill | 检查路径 | 安装命令 |
|-------|---------|---------|
| doc-reader | `.cursor/skills/doc-reader/` | `curl -sL "http://jueying.corp.kuaishou.com/api/admin/skill/install-script?name=doc-reader&scope=project" | bash` |
| rules-generator | `.cursor/skills/rules-generator/` | `curl -sL "http://jueying.corp.kuaishou.com/api/admin/skill/install-script?name=rules-generator&scope=project" | bash` |

### 2. Rules 检查

1. **严格检查项目 Rules 是否存在**：
   - **必须检查文件**：`.cursor/rules/templates/base/project-base-rules.mdc`
   - **注意**：文件名必须严格匹配 `project-base-rules.mdc`，不能是 `web-base-rules.mdc` 或其他名称
   - 使用 `search_file` 工具精确搜索：`query=".cursor/rules/templates/base/project-base-rules.mdc"`

2. **如果 Rules 不存在，调用 rules-generator skill 生成**：
   ```
   使用 use_skill 工具调用：
   - skill_name: "rules-generator"
   - reason: "项目缺少 project-base-rules.mdc，需要先生成项目基础 Rules 才能继续分析架构"
   ```

3. **读取所有 Rules**：
   - `.cursor/rules/templates/**/*.mdc`

### 3. 准备输出路径

```bash
# 获取分支名
BRANCH=$(git rev-parse --abbrev-ref HEAD | tr '/' '-')
OUTPUT_DIR="architect-docs/${BRANCH}"

# 如果目录已存在，在后面加序号
if [ -d "$OUTPUT_DIR" ]; then
  i=1
  while [ -d "${OUTPUT_DIR}_${i}" ]; do
    i=$((i+1))
  done
  OUTPUT_DIR="${OUTPUT_DIR}_${i}"
fi

mkdir -p $OUTPUT_DIR
```

输出文件：
- `architect-docs/[分支名]/architecture.md`
- `architect-docs/[分支名]/tasks.md`

如果目录已存在，则加序号：`architect-docs/[分支名]_1/`

### 4. 代码结构分析

使用 AI IDE 工具分析项目：
- `codebase_search`：搜索代码结构
- `read_file`：读取 `package.json`、现有页面文件
- `list_dir`：查看 `src/pages/`、`src/api/` 目录结构

### 5. 生成架构文档并保存

## 分析要求

### 1. PRD 内容提取（⚠️ 关键步骤）

**必须详细提取 PRD 中的所有业务信息**：

1. **页面布局信息**：
   - 每个页面的筛选条件（字段名、类型、是否必填、选项值等）
   - 表格列定义（列名、字段、宽度、渲染方式、是否固定等）
   - 操作按钮（位置、文案、交互逻辑）
   - 页面布局结构（头部、内容区、弹窗等）

2. **字段定义**：
   - 所有数据字段的详细说明（字段名、类型、格式、展示方式）
   - 枚举值定义（状态值、类型值等）
   - 字段之间的关联关系

3. **弹窗/表单定义**：
   - 每个弹窗的字段列表
   - 字段的验证规则
   - 弹窗的交互流程
   - 提交逻辑和状态流转

4. **业务规则**：
   - 状态流转规则（如：归因状态 → 修复状态）
   - 权限控制规则
   - 数据校验规则
   - 特殊交互逻辑（hover 提示、条件显示等）

5. **图片分析**（如果有）：
   - 使用 `read_file` 读取 `.cursor/skills/doc-reader/temp/images/` 目录下的所有图片
   - **识别图片类型**：
     - 页面示意图：包含完整页面布局、表格、表单等 UI 元素
     - 流程图：展示业务流程、状态流转
     - 其他：弹窗截图、细节说明图等
   - **针对页面示意图**：
     - 分析图片中的 UI 布局、字段位置、交互元素
     - 提取图片中的表格列、表单字段、按钮等信息
     - 识别页面结构（头部、筛选区、表格区、操作区等）
   - **针对流程图**：
     - 提取业务流程步骤
     - 识别状态转换节点
     - 提取条件判断逻辑

### 2. 代码分析

使用 AI IDE 工具分析现有代码：
- 分析现有页面、组件、API 的组织方式
- 识别可复用的组件、工具函数、hooks
- 判断需求类型：新增页面/功能 or 修改现有功能

### 5. 开发任务列表

**⚠️ 重要原则**：
- 任务列表是给 AI Coding 看的 TODO List，**不要生成代码**
- 清晰描述要做什么、创建什么文件、需要什么逻辑
- 方便后续 AI 根据任务列表逐个实现
- 方便检查功能完成度

**任务表格格式**：

| 任务ID | 任务描述 | 优先级 | 文件路径 | 依赖 |
|-------|---------|-------|---------|-----|
| T1 | [简洁描述] | P0/P1/P2 | [文件路径] | 无 |
| T2 | [简洁描述] | P0/P1/P2 | [文件路径] | T1 |

**优先级说明**：
- **P0**：核心功能，必须实现（API、类型定义、页面主文件、路由）
- **P1**：重要功能，影响用户体验（组件、弹窗、关键交互）
- **P2**：次要功能，锦上添花（样式优化、辅助功能）

**任务分层结构**（建议按层级组织任务）：

```
1. API 接口层（T1-T3）
   ↓
2. 类型定义层（T4-T6）
   ↓
3. 页面主文件层（T7-T9）
   ↓
4. 组件层（T10-T30）
   ↓
5. 样式文件层（T31-T33）
   ↓
6. 路由配置层（T34）
   ↓
7. 测试验证层（T35-T36）
```

**关键任务详细说明格式**：

- `update[XXX]`: 编辑
- `delete[XXX]`: 删除

**关键参数说明**：
- 查询接口：分页参数、筛选条件

**实现要点**：

**任务列表生成要点**：

1. **任务颗粒度**：
   - API 文件：每个模块一个任务
   - 类型定义：每个页面一个任务
   - 页面主文件：每个页面一个任务
   - 组件：每个组件一个任务
   - 样式文件：每个页面一个任务

2. **任务描述**：
   - 使用动词开头：创建、实现、添加、修改
   - 简洁明了，一句话说清楚
   - 避免冗余描述

3. **详细说明**：
   - 只对**关键任务**提供详细说明（API、页面主文件、复杂组件）
   - 使用 TODO List 形式列举功能点
   - 列出字段、参数、逻辑要点
   - **不要写完整代码**，只描述要做什么

4. **代码复用指引**：
   - 优先使用项目现有的组件、工具、hooks
   - 但**核心业务逻辑必须严格遵循 PRD**，而不是照搬参考代码
   - 参考代码只用于学习项目代码风格和技术栈使用方式
   - 如果 PRD 与参考代码不一致，**以 PRD 为准**

5. **依赖关系**：
   - 明确任务依赖，确保开发顺序正确
   - API 和类型定义优先，页面次之，组件最后

## 约束

1. **只关注前端和本仓库工作**，后端/其他团队工作一笔带过
2. **优先复用**：优先使用项目现有组件、工具，避免重复造轮子
3. **详细完整**：架构文档必须包含足够的业务上下文，让 AI Coding 能够直接生成代码
4. **必须提取 PRD 细节**：
   - 所有字段定义必须从 PRD 中提取
   - 所有表格列必须从 PRD 中提取
   - 所有弹窗字段必须从 PRD 中提取
   - 所有状态流转规则必须从 PRD 中提取
   - 所有图片中的 UI 信息必须分析提取（需识别图片类型，重点分析页面示意图）
5. **标注不确定项**：不明确的信息标注为"待确认"
6. **不要写**：开发计划、风险评估、单元测试、工作量估算、开发进度
7. **严格遵循 PRD，不是参考代码**：
   - 任务描述必须基于 PRD 中的业务需求，而不是参考代码的实现方式
   - 即使有参考代码路径，**核心字段、交互逻辑必须严格按照 PRD 来**
   - 如果 PRD 和参考代码不一致，**以 PRD 为准**，可以在任务中标注"与参考代码不同"
   - 组件功能必须从 PRD 的弹窗/表单定义中提取，而不是从参考代码推断
